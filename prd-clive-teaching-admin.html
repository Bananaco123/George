第一章项目概述

本章说明 Clive 教学管理后台的产品定位、阶段目标与核心指标，以及服务范围与参与角色，为后续需求描述提供业务背景。

1.1产品背景

Clive 是汉韵无界工作室内部的直播教学项目代号（Chinese Live 的缩写），面向东南亚中文教育市场，首期聚焦泰国市场，采用 B2B 模式与当地学校合作。首期目标（MVP）：在 2026 年 6 月前为合作学校的师生提供「在线远程直播 → 课后直播回放」的完整在线教学服务闭环。

教学管理后台（Admin Ops）是 Clive 平台的运营中台，仅供汉韵工作室内部运营团队和内部授课老师使用，承担以下核心职责：

  • 基础数据初始化：维护国家、年级标准、学科等系统码表
  • 学校组织管理：创建并维护合作学校的组织结构（学校 / 班级 / 老师 / 学生账号）
  • 课程体系搭建：创建课程与课时，配置直播间，实现排课与发布
  • 直播资源管理：管理直播间和录播视频
  • 权限管控：基于 RBAC 管理内部用户账号与角色权限
  • 工作台：为内部授课老师提供课时日历视图和进入直播间的专属入口
范围说明：本文档仅描述教学管理后台（Admin Ops）的需求。学校管理员、老师和学生所使用的功能页属于「用户 Web 端」（另立 PRD），不在本文档范围内。

2.3账户体系设计

Clive 项目账户体系分两个阶段演进，以满足不同阶段的业务形态：

第一阶段：ToB 学校交付（当前阶段）

第一阶段以手动创建为核心，由汉韵无界工作室运营团队在教学管理后台统一创建所有账号，包括：内部运营人员、内部授课老师，以及各合作学校的老师和学生账号。

账户体系严格分为两套相互独立的用户体系，两套账号的数据表、登录入口和权限体系完全分离，互不影响：

  • 内部用户体系（InternalUser）：供汉韵工作室内部人员使用，登录入口为教学管理后台；角色包括超级管理员、管理员、内部授课老师；账号由超级管理员手动创建
  • 学校用户体系（Teacher / Student）：供合作学校的老师和学生使用，登录入口为用户 Web 端；账号由运营团队在管理后台手动创建，支持 Excel 批量导入
重要设计原则：学校老师账号（Teacher）与内部授课老师账号（InternalUser with INTERNAL_LECTURER role）是两种完全不同的账号类型。学校老师登录用户端，仅能查看所带班级课表；内部授课老师登录管理后台，仅能访问工作台并进入直播间。两者不可混用。

  2. 用户已知自己的登录账号（username）与密码
后置条件
  1. 用户成功登录，进入教学管理后台主界面
  2. 若用户拥有 workbench 权限，默认进入工作台页；否则进入系统默认首页（国家管理）
  3. 登录态通过 localStorage 持久化，刷新页面后自动恢复，无需重复登录
交互原型 admin-ops-v1.html → 登录页（打开即见）
主事件流 1 初始化超级管理员：
  1. 系统首次部署时，需手动创建超级管理员账号，用于后续登录后台进行系统配置
  2. 初始化超级管理员账号信息： 用户名（username）：superadmin 初始密码：123456 角色：超级管理员（拥有全部功能权限 *） 状态：启用
  3. 初始化方式：通过数据库脚本或系统初始化接口创建该账号记录
  4. 超级管理员首次登录后，建议立即修改初始密码（密码修改功能在第二阶段实现）
主事件流 2 内部用户登录：
  1. 用户打开教学管理后台 URL，系统读取 localStorage 中保存的会话信息
  2. 若无有效会话（未登录 / 会话失效），展示登录页：居中卡片布局，包含账号输入框、密码输入框（支持明文/密文切换）、登录按钮
  3. 用户填写账号与密码，点击「登录」按钮（或在密码输入框按 Enter 触发）
  4. 系统依次执行校验： ① 账号与密码均不为空 ② 账号在用户列表中存在 ③ 密码与该账号记录匹配 ④ 账号状态为「启用」
  5. 全部校验通过后，将 userId、username 写入 localStorage，完成登录
  6. 系统读取当前用户的权限集合： 若包含 workbench 权限 → 跳转工作台 否则 → 跳转第一个有权限的菜单页（因为每个角色不一定有国家管理页权限）
  7. 顶部导航栏右侧显示当前登录的用户名（username），并提供「退出登录」入口
其他事件流 1 刷新页面 / 重新打开时自动恢复登录态：
  1. 页面加载时，读取 localStorage 中保存的 userId 与 username
  2. 以 userId 在用户列表中查找状态为「启用」的匹配账号
  3. 若找到，直接恢复登录态，跳过登录页，进入后台主界面
  4. 若未找到，退一步以 username 再次匹配；若匹配成功，更新 userId 后持久化并恢复登录
  5. 若两次匹配均失败，清除 localStorage，跳转回登录页
其他事件流 2 退出登录：
  1. 用户点击顶部导航「退出登录」按钮
  2. 系统清除 localStorage 中的会话信息，清空内存 auth 状态
  3. 页面跳转回登录页，表单恢复初始空白状态
异常流 1 账号或密码为空：点击「登录」按钮后，系统提示「请输入账号和密码」，不触发任何匹配逻辑，输入框保持当前状态，等待用户补充输入。
异常流 2 账号/密码不匹配，或账号已被关闭：系统统一返回提示「账号或密码错误，或账号已被关闭」（不区分具体原因，防止账号枚举攻击），密码输入框内容清空，账号框保留用户输入，用户可直接重新输入密码重试。
备注
  • 原型演示中登录页预填账号 superadmin / 123456，方便快速体验，正式上线前须移除预填值
  • 正式实现时密码须加密存储（推荐 bcrypt 或 argon2），禁止明文存入数据库
  • 用户数据中 userId 必须为稳定固定值（如 iu-superadmin），禁止在每次页面初始化时动态生成 uuid 作为种子数据，否则页面刷新后 userId 变化将导致已登录会话无法恢复，被强制退出
  • 第二阶段考虑接入第三方 OAuth（LINE / Google），当前第一阶段仅支持账号密码方式

权限并集规则
一个内部用户可同时绑定多个角色。用户的实际可操作权限 = 各绑定角色的 permission_keys 的并集（Union）。例如：用户绑定角色 A（拥有 course:manage）和角色 B（拥有 live:manage），则该用户同时拥有两个权限。
若任一绑定角色的权限集合包含 *，则该用户拥有系统全部功能权限，与其他角色绑定无关。

异常流 2 角色名称为空：系统提示「请填写角色名称」，禁止提交。
备注
  • 权限项为空也可保存（表示该角色暂无任何功能权限，仅作占位）
  • 系统内置角色（SUPER_ADMIN / ADMIN / INTERNAL_LECTURER）不支持删除，但可编辑权限项（超管角色的 * 权限建议锁死，不允许修改）

依赖风险提示：A8 直播间的创建、进房 Token 生成、直播状态回调均依赖 LiveProvider（直播服务供应商）接入完成。若供应商在 Step 2 期间尚未确认，A8-1-1 创建直播间和 A8-1-3 直播间列表的进房操作将被阻塞，建议先完成供应商选型。
A8-1-1  创建直播间
参与者 超级管理员 / 管理员（需有 live:manage 权限） 优先级 P0
前置条件
  1. 操作者已登录，拥有 live:manage 权限
  2. LiveProvider（如声网灵动课堂）已完成对接配置，API 可正常调用
  3. 系统中已有内部授课老师账号（角色含 INTERNAL_LECTURER），用于选择授课老师
后置条件
  1. 系统创建 LiveSession 记录，调用 LiveProvider API 完成房间分配，获得房间 ID 和进房凭证
  2. 直播间出现在直播列表，状态为「未开始」
  3. 该直播间可在课时管理（A10-4-1）中被引用，关联到具体课时
交互原型 admin-ops-v1.html → 直播管理 → 直播列表 → 新增直播间
主事件流
  1. 进入「直播管理 → 直播列表」，点击「新增直播间」按钮
  2. 弹出「新增直播间」对话框，包含以下字段： 直播间标题（必填，展示给学生的直播名称） 内部授课老师（必选，从内部用户中筛选角色含 INTERNAL_LECTURER 的账号） 计划开始时间（必填，日期时间选择器，精确到分钟） 计划时长（必填，分钟数，正整数，如 90） 直播模式（单选：大班课 / 小班课，影响 LiveProvider 的房间类型参数） 最大参与人数（可选，超出后学生无法进入直播间） 备注（可选，内部备注，学生不可见）
  3. 点击「保存」，校验必填项
  4. 校验通过后，系统调用 LiveProvider API 创建直播房间，获取 room_id 和进房 Token
  5. LiveSession 记录写入数据库，直播间列表刷新，提示「直播间已创建」
异常流 1 必填项为空：提示对应字段不可为空，禁止提交。
异常流 2 LiveProvider API 调用失败：提示「直播间创建失败，请检查网络连接或联系技术支持」；LiveSession 记录不写入数据库，不计入列表。操作者可重试。
备注
  • 直播间的创建与具体课程无直接耦合，直播间是独立资源，可被多个课时引用（A10-4-1 新增课时时从直播间列表选择）
  • 计划开始时间不代表直播间实际开启时间，授课老师可提前 15 分钟进入准备，学生在计划时间前后均可尝试进入
  • 进房 Token 由 LiveProvider 控制有效期，系统需在每次进房请求时动态生成或续期，而非固定存储

A10-4-4  上架 / 下架课时

3、下架操作的影响范围：

A10-10  编辑课程
参与者 超级管理员 / 管理员（需有 course:manage 权限） 优先级 P0
前置条件 目标课程已存在（草稿 / 已发布 均可进入编辑页）
后置条件 课程信息按保存规则更新；已发布课程中需手动保存的内容点击「保存课程」后对学生实时生效
交互原型 admin-ops-v1.html → 课程列表 → 编辑课程（行操作）
主事件流

1、在课程列表中点击「编辑课程」，进入课程编辑页（与新建课程同一页面，数据回显该课程当前内容）

2、课程编辑页包含四个内容区域：基础信息（A10-2）、课程介绍（A10-3）、课时管理（A10-4）、课程资源（A10-5）

3、发布前（草稿）编辑规则

  • 课程状态标识显示「草稿」，操作按钮显示「保存草稿」
  • 所有内容区域均可自由编辑，无任何限制
  • 修改基础信息和课程介绍后，须点击「保存草稿」才能持久化；课时和资源操作在对话框内单独保存
  • 草稿课程不对学生可见，任何修改不影响学生端
  • 满足发布条件后，右上角出现「发布课程」按钮（详见 A10-7）

4、发布后（已发布）编辑规则

  • 课程状态标识显示「已发布」，操作按钮显示「保存课程」
  • 所有内容区域均可继续编辑，编辑后课程状态不会自动回退为草稿

各类修改的生效方式如下：

修改内容生效方式对学生端影响
基础信息（学科 / 课程名 / 类型）须点击「保存课程」保存后学生下次进入课程页即看到新内容
课程介绍（富文本）须点击「保存课程」保存后学生下次访问即看到新介绍
上架课时操作后立即生效学生端立即可见该课时入口
下架课时操作后立即生效学生端立即不可进入该课时（不影响已在课时页的学生）
新增 / 编辑课时课时弹窗内点「保存」后立即写入已上架课时变更学生端立即可见
课外资源上传 / 删除操作后立即写入学生端同步变更
备注
  • 保存课程的完整逻辑（按钮文案、保存时机、实时生效规则）详见 A10-6
  • 编辑已发布课程不会使课程状态回退为草稿；如需重新配置发布范围，使用 A10-13「调整发布范围」

A10-12  下架课程
参与者 超级管理员 / 管理员（需有 course:manage 权限） 优先级 P0
前置条件 目标课程当前状态为「已发布」；草稿课程不显示「下架课程」操作入口
后置条件 课程状态变为「草稿」，学生端立即不可见该课程；各课时的上架 / 待上架状态不变
交互原型 admin-ops-v1.html → 课程列表 → 下架课程（行操作，仅已发布课程显示）
主事件流

1、在课程列表中，点击已发布课程行操作区的「下架课程」按钮

2、弹出二次确认弹窗：

  • 弹窗标题：「下架确认」
  • 弹窗内容：「确定下架课程「{课程中文名}」？下架后课程为草稿、学员侧不可见。课时的「上架/待上架」状态不会改变，再次发布时无需因课程下架而重新操作课时上下架。」
  • 按钮：「取消」（关闭弹窗，不执行下架）、「确定下架」（执行下架操作）

3、操作者点击「确定下架」后：

  • 课程状态变为「草稿」，学生端立即不可见该课程
  • 课程下各课时的上架 / 待上架状态保持不变，不会被重置
  • 提示「已下架课程；课时上架状态已保留，满足条件时可再次发布」
  • 课程列表刷新，对应课程状态标签变为「草稿」
备注
  • 下架后课程不会自动清空发布范围配置（国家、区域等），再次发布时可直接在原配置基础上修改
  • 下架后若课程满足发布条件（基础信息完整 + 课程介绍非空 + 至少有一个已上架课时），可直接再次点击「发布课程」重新发布（详见 A10-7）
A10-13  调整发布范围
参与者 超级管理员 / 管理员（需有 course:manage 权限） 优先级 P0
前置条件 目标课程当前状态为「已发布」；草稿课程不显示「调整发布范围」操作入口
后置条件 课程发布范围更新，新的发布范围立即生效；课程状态保持「已发布」不变
交互原型 admin-ops-v1.html → 课程列表 → 调整发布范围（行操作，仅已发布课程显示）
主事件流

1、在课程列表中，点击已发布课程行操作区的「调整发布范围」按钮

2、在当前页面弹出「调整发布范围」弹窗（与发布课程弹窗相同结构，详见 A10-7 发布范围弹窗字段说明）

3、弹窗回显该课程当前的发布范围配置：

  • 「发布国家」预填当前已选国家
  • 「区域收窄」预填当前已选省市
  • 「额外开放学校」预填当前已配置学校
  • 「排除学校」预填当前已配置学校

4、所有字段均支持修改（新增 / 删除选项）

5、点击「保存」，弹出二次确认弹窗：

  • 弹窗标题：「保存范围确认」
  • 弹窗内容：「确定保存当前发布范围？」
  • 按钮：「取消」（返回范围弹窗，不保存）、「保存」（执行保存）

6、操作者点击「保存」后：

  • 新的发布范围写入数据库，立即生效
  • 课程状态保持「已发布」，不会因调整范围而下架
  • 提示「发布范围已更新」，弹窗关闭
异常流
  • 清空全部发布国家：点击「保存」时提示「请至少选择一个发布国家」，弹窗不关闭
备注 调整发布范围不影响课程的发布状态，也不触发重新发布流程；若需完全重新发布，须先执行「下架课程」（A10-12）再重新发布（A10-7）

A11-1  上课工作台
参与者 内部授课老师 / 管理员（需有 workbench 权限） 优先级 P0
前置条件 操作者已登录，拥有 workbench 权限；登录后系统默认进入「工作台」页面

数据范围

工作台展示的课时数据，需同时满足以下全部条件：

  • 所属课程状态为 「已发布」（course.status = 'published'）
  • 课时状态为 「已上架」（lesson.status = 'on_shelf'）
  • 课时类型为 「直播」（lesson.lesson_kind = 'live'）
  • 课时的 授课老师 = 当前登录内部用户（lesson.internal_teacher_id = auth.userId）
  • 课时有合法的 开始时间（start_time 字段已填写）

以上数据范围同时作用于月历圆点标记、底部统计数据、右侧直播课时列表。
主事件流 1
月历视图

1、页面布局

  • 工作台采用左右两栏布局：左侧为月历视图，右侧为直播课时列表
  • 进入工作台时，默认选中自然日的今天，右侧立即显示今天的直播课时

2、月历顶部导航

  • 显示当前月份标题「YYYY 年 MM 月」
  • 左侧「‹」按钮切换上月，右侧「›」按钮切换下月
  • 切换月份后，若当前选中日期不在新月份内，自动将选中日期改为新月份的第一天（YYYY-MM-01）

3、月历日期格

  • 月历按周排列（周一至周日），每格显示日期数字
  • 自然日「今天」在日期数字上显示特殊高亮（如「今」标记）
  • 当前选中日期以蓝色加粗样式区分

4、日期底部圆点（仅在数据范围内有课时时显示）

（1）课时结束时间计算规则

  • 课时结束时间 = 课时开始时间 + 课时时长
  • 若课时时长字段为空或无效，则以「当前时间是否已超过课时开始时间」作为结束判断依据

（2）已结束判定规则

  • 当前实时时间 ≥ 课时结束时间，则该课时判定为「已结束」
  • 判断使用当前实时时间（约每秒刷新）

（3）圆点显示规则

当天情况圆点显示规则
当天没有直播课时不显示圆点
当天有直播课时，且存在尚未结束的课时（当前时间 < 课时结束时间）显示红点
当天有直播课时，且全部已结束（当前时间 ≥ 全部课时的结束时间）显示绿点
  • 圆点尺寸约 6px，显示于日期数字下方，不遮挡日期数字

5、点击日期格

  • 点击月历中任意日期格后，该格切换为选中状态（蓝色加粗）
  • 右侧直播课时列表联动更新为该选中日期的课时，详见「主事件流 2」

6、月历底部统计数据

月历下方以三栏形式展示统计数字，统计范围均取自上方「数据范围」内的课时：

统计项统计规则显示规则
当天课时数 当前选中日期（非固定自然日「今天」）下，符合数据范围的直播课时数量 随月历选中日期实时联动更新；无课时时显示 0
当月课时数 当前月历所在月份（YYYY-MM）内，符合数据范围的直播课时数量 切换月份后更新；无课时时显示 0
课程总课时数 符合数据范围的全部直播课时数量（不限月份、不限日期） 固定统计值，不随日期/月份切换变化
主事件流 2
直播课时列表

1、列表标题与节数

  • 标题格式：「直播课时列表 · M月D日」（如「直播课时列表 · 4月14日」），随选中日期更新
  • 标题右侧显示 Tag：「共 N 节直播」（N = 选中日期下符合数据范围的课时数）

2、列表字段

字段名字段类型说明
